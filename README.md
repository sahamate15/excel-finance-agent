# 💹 Excel Finance Automation Agent

A modular Python agent that turns plain-English finance instructions into Excel
formulas and multi-row models. Built for analysts who'd rather describe a
depreciation schedule than wire it up cell by cell.

---

## Features

- **Formula generation** — "growth rate between B2 and B3" → `=(B3-B2)/B2`
- **Table builders** — depreciation, loan amortization, multi-year revenue
  projection, all wired with live Excel formulas (never hardcoded values)
- **Hybrid resolution** — a deterministic fallback map handles common requests
  instantly; the LLM kicks in for the ambiguous ones
- **Both interfaces** — Rich-styled CLI (`python main.py`) and Streamlit UI
  (`streamlit run app.py`)
- **Industry-standard styling** — blue inputs, black formulas, alternating row
  fills, currency / percent formats applied automatically
- **Robust** — full validation, retry loop on malformed LLM output, rotating
  log file at `logs/agent.log`

## Setup

**1. Python 3.10 or newer**

```bash
python --version
```

**2. Install dependencies**

```bash
pip install -r requirements.txt
```

**3. Configure environment**

```bash
cp .env.example .env
# Edit .env and paste your Mistral API key
```

Get a free-tier key at <https://console.mistral.ai/api-keys/>.

Minimum viable `.env`:

```
MISTRAL_API_KEY=...
MISTRAL_MODEL=mistral-small-latest
DEFAULT_EXCEL_FILE=data/sample_workbook.xlsx
DEFAULT_SHEET=Revenue
FALLBACK_FIRST=true
```

### Why Mistral?

The agent talks to Mistral's chat completions endpoint via its
OpenAI-compatible API (`https://api.mistral.ai/v1`), so the standard `openai`
Python SDK is reused as a thin client — no second dependency.

| Model                    | Use case                                    | Cost          |
| ------------------------ | ------------------------------------------- | ------------- |
| `mistral-small-latest`   | Default — fast, cheap, sufficient for finance formulas and JSON config extraction | Free tier     |
| `mistral-medium-latest`  | Slightly stronger reasoning for ambiguous instructions | Paid          |
| `mistral-large-latest`   | Highest quality for unusual or multi-step instructions | Paid          |

Override with `MISTRAL_MODEL=mistral-medium-latest` (etc.) in `.env`. The
hardcoded fallback formula map and keyword-based task classifier still run
without any API key, so the agent remains usable offline for common
requests.

**4. Generate the sample workbook**

```bash
python generate_sample_workbook.py
```

This creates `data/sample_workbook.xlsx` with three pre-loaded sheets
(Revenue, Assets, CashFlows) used in the examples below.

```bash

```

```bash

```

## Example instructions

### Formula requests

| # | Instruction                                                       | Sample output           |
| - | ----------------------------------------------------------------- | ----------------------- |
| 1 | calculate growth rate between B2 and B3                           | `=(B3-B2)/B2`           |
| 2 | what is the CAGR from B2 to B6 over 4 years                       | `=(B6/B2)^(1/4)-1`      |
| 3 | calculate IRR for cash flows in B2:B7                             | `=IRR(B2:B7)`           |
| 4 | NPV at 12% discount rate for B3:B7                                | `=NPV(0.12,B3:B7)`      |
| 5 | EMI for 50 lakh loan at 8.5% for 20 years                         | `=PMT(0.085/12,240,-5000000)` |
| 6 | net profit margin if revenue is in B2 and profit in C2            | `=C2/B2`                |
| 7 | WDV depreciation for year 3 on asset value in B2 at 25%           | `=ROUND(B2*(1-0.25)^3,2)` |

### Table requests

| #  | Instruction                                                                  |
| -- | ---------------------------------------------------------------------------- |
| 8  | create a 5-year WDV depreciation schedule for 20 lakh at 25%                |
| 9  | build loan amortization table for 50 lakh at 9% for 10 years                |
| 10 | make a 5-year revenue projection starting at 50 lakh with 15% growth        |
| 11 | create EBITDA table for years 2020–2024 using Sheet Revenue data            |

For tables, the agent writes the schedule into the active sheet starting at
A1 (or the position you supply via the LLM's parsed config) with industry-
standard styling: blue inputs, black formulas, bordered alternating rows.

## How it works

1. **Sanitise** the input (`utils/validators.sanitize_instruction`)
2. **Classify** with the LLM as `formula` / `table` / `chart`; keyword fallback
   if the API is unreachable
3. **Clarify** if essential inputs are missing — the agent returns a
   single-question clarification request rather than guessing
4. **Resolve**:
   - For formulas: try the deterministic fallback map first (when
     `FALLBACK_FIRST=true`), then the LLM with a tightly-scoped system prompt
     and validation + 2 retries
   - For tables: ask the LLM for a `table_config` JSON, then dispatch to the
     matching builder in `excel_writer.py`
5. **Write** with openpyxl, applying styling and saving the workbook
6. **Log** every step to `logs/agent.log` (5 MB rotating, 3 backups)

## Troubleshooting

**`MISTRAL_API_KEY is not set`** — Add it to `.env` (see step 3 above) or
export it in your shell: `export MISTRAL_API_KEY=...`

**`Default workbook does not exist`** — Run
`python generate_sample_workbook.py` once to create it.

**Streamlit shows "No workbook loaded"** — Either generate the sample
workbook or upload your own `.xlsx` via the sidebar.

**Formulas show as text in Excel** — openpyxl writes formulas but doesn't
evaluate them. Open the file in Excel/LibreOffice once and they'll calculate
on first open. (For headless recalculation, install LibreOffice and run
`soffice --headless --calc --convert-to xlsx file.xlsx`.)

**LLM returns invalid output** — The agent retries up to `MAX_RETRIES`
(default 2) times. If all retries fail it falls back to the offline map; if
that also has no match, you'll get a `ValueError` with the last LLM error
message. Rephrase the instruction more concretely (include cell references)
and try again.

**`logs/agent.log` is empty** — Check `LOG_LEVEL` in `.env`; set it to
`DEBUG` for verbose tracing.

**Tests skip the LLM cases** — That's expected without an API key. Set
`MISTRAL_API_KEY` and rerun to include them.

## Audit and compliance

Every action the agent takes is recorded as a structured event in an
append-only, tamper-evident audit log. Designed for investment-banking
compliance review, where reviewers need to know *exactly* what the agent
did to a deal model — without that record itself becoming a source of
deal-data leakage.

### What is logged

- **Session lifecycle**: session start / end, mode change (strict ↔ AI),
  API key loaded / cleared, model selection.
- **File events**: file uploaded (filename, size, SHA-256 hash of contents),
  sheet selected, file downloaded.
- **Instruction events**: the raw (sanitized) instruction text, the
  detected task type, and whether classification came from the LLM or the
  offline keyword classifier.
- **Formula / table events**: every formula generated, dry-run preview
  shown, dry-run confirmed / cancelled, formula written, formula write
  failed, table config extracted, table built, table build failed.
- **Validation events**: input rejections (with field name and reason),
  formulas rejected by the syntactic validator.
- **Error events**: any unhandled exception with the action that triggered it.

### What is NOT logged

**Cell values from the workbook are NEVER written to the audit log.**
Only formulas (what was written), target locations (file basename, sheet,
cell), the instruction text the user typed, and SHA-256 hashes of file
contents are recorded. This preserves the audit trail while keeping
proprietary deal data out of a separate log file that may have different
access controls.

### Where it lives

Daily JSONL files at `logs/audit/YYYY-MM-DD.jsonl` (UTC date). One file
per day, append-only, never overwritten. Each line is one event:

```json
{
  "ts":         "2026-04-29T10:15:23.451Z",
  "session_id": "<uuid>",
  "seq":        42,
  "event_type": "formula_written",
  "payload":    {"file": "deal_model.xlsx", "sheet": "Revenue",
                 "cell": "F12", "formula": "=AVERAGE(B2:B100)",
                 "source": "offline_fallback"},
  "prev_hash":  "sha256:..."
}
```

### Tamper-evidence

Each event carries the SHA-256 of the previous event's full JSON line
(within the same session, in the same daily file). The first event of a
session in a given file has `prev_hash: null`. Any silent edit to a
persisted line breaks the hash chain on the next event, which the
verification function will report.

This is **not bulletproof** against an attacker who rewrites the entire
chain, but it makes silent edits impossible and casual tampering obvious.
For a stronger threat model, deploy on WORM (write-once-read-many) storage
— that integration is out of scope here.

### Querying

A small CLI ships with the project:

```bash
# All formula writes for one file in a date range
python audit_query.py --from 2026-04-01 --to 2026-04-29 \
    --event-type formula_written --file deal_model.xlsx

# Everything one session did, as CSV
python audit_query.py --session-id <uuid> --format csv > session.csv

# Check the chain for a given day
python audit_query.py --verify 2026-04-29
```

Filters: `--from / --to`, `--session-id`, `--event-type` (repeatable),
`--file`, `--file-hash`, `--source`. Output formats: `table` (default),
`json`, `csv`. The `--verify` subcommand exits 0 if the chain is intact,
1 if any breaks were found.

The Streamlit UI exposes the same capabilities under the **Audit Log**
tab — date picker, multi-select event-type filter, file substring filter,
source filter, CSV export of filtered rows, and a **Verify integrity**
button that runs the chain check on the displayed day.

### Retention and rotation

Audit logs **never auto-delete**. After 90 days, daily files older than
that are flagged in `logs/audit/archive_manifest.json` (call
`utils.audit.write_archive_manifest()` to regenerate); the firm's IT can
plug that manifest into their own archival pipeline. The agent itself
never moves or deletes audit files.

`logs/audit/.gitignore` excludes everything in the audit directory from
version control as a defence-in-depth alongside the project-root
`.gitignore`.

## Code-quality notes

- Python 3.10+, type hints on every public signature, Google-style docstrings
- No bare `except` clauses, no global mutable state outside of explicit
  singletons (`CONFIG`, LLM client, logger)
- `pathlib.Path` for all file operations, context managers everywhere
- Every monetary cell uses Excel number formats, not Python string formatting
- Industry-standard finance colour conventions: **blue** for inputs,
  **black** for formulas, **green** for cross-sheet links, blue header fill
  on table headers
